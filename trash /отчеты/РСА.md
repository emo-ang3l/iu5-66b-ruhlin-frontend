# РУКОВОДСТВО СИСТЕМНОГО АДМИНИСТРАТОРА  
распределённой информационной системы (прикладной, транспортный и агентный уровни)

Документ предназначен для специалиста, ответственного за **развёртывание**, **конфигурирование** и **сопровождение** компонентов системы на учебном стенде (три узла локальной сети или консолидированный хост).

---

## 1. Общие сведения

Система состоит из трёх логических уровней:

1. **Прикладной** — клиент React + сервер Node.js (Express, ws), порт по умолчанию **8001**, Swagger UI **`/api-docs`**.  
2. **Транспортный** — сервис приёма сегментов, брокер **Apache Kafka**, контейнеризация **Docker** (рекомендуется методичка кафедры).  
3. **Агентный** — сервис на **Python FastAPI** (или согласованный стек), периодическая генерация телеметрии варианта 6, HTTP-клиент к транспорту.

## 2. Системные требования по компонентам

### 2.1. Узел прикладного уровня

| Ресурс | Минимум | Рекомендуется |
|--------|---------|---------------|
| ОС | Windows 10/11, Linux, macOS | Linux для стенда |
| Node.js | LTS 18+ | LTS 20+ |
| RAM | 2 ГБ свободно | 4 ГБ |
| Диск | 500 МБ | 2 ГБ |

Сервис прикладного уровня использует зависимости из `WebSocketServer/package.json`; клиентская часть — `package.json` в корне репозитория.

### 2.2. Узел транспортного уровня

| Ресурс | Минимум | Рекомендуется |
|--------|---------|---------------|
| ОС | Linux x86_64 | то же |
| Docker / Compose | актуальная стабильная | — |
| RAM | 4 ГБ (с Kafka) | 8 ГБ |
| Диск | 10 ГБ | SSD |

Требуется открыть порты для **Kafka** (например, 9092) и для HTTP API транспортного сервиса (порт команды, по методичке часто **8002** — проверьте фактический).

### 2.3. Узел агентного уровня

| Ресурс | Минимум | Рекомендуется |
|--------|---------|---------------|
| Python | 3.10+ | 3.11+ |
| RAM | 1 ГБ | 2 ГБ |

При использовании виртуальной машины с **ROS 2** требования увеличиваются (CPU 2+ ядра, RAM 4+ ГБ) — по выбору команды.

## 3. Сетевая конфигурация

- На всех узлах зафиксировать **статические IP** или резервировать адреса в DHCP.  
- В файле `WebSocketServer/index.ts` задать `transportLevelHostname` и `transportLevelPort` равными фактическому адресу транспортного сервиса (в репозитории по умолчанию указаны учебные значения — **заменить** перед демонстрацией).  
- Клиентский frontend должен знать хост WebSocket (переменные окружения Vite или конфиг сборки).  
- При использовании **ZeroTier** следуйте методичке кафедры: объедините узлы в одну логическую сеть, проверьте **ping** между хостами.

## 4. Развёртывание прикладного уровня

```bash
git clone <URL-репозитория>
cd iu5-66b-ruhlin-frontend
npm install
npm run dev
```

В отдельном терминале:

```bash
cd WebSocketServer
npm install
npx tsc
node index.js
```

Убедитесь, что сервер слушает **`http://localhost:8001`** (или заданный хост). Проверка: открыть **`http://localhost:8001/api-docs`**.

---

## 5. Развёртывание транспортного уровня

Порядок действий зависит от репозитория команды. Общий чек-лист:

1. Поднять **Kafka** и **Zookeeper** (или KRaft — по версии стека) через Docker Compose.  
2. Запустить бинарник/контейнер сервиса транспорта.  
3. Настроить переменные: адрес **Kafka**, **N** (период сборки), **R** (процент потери), URL **`POST /receive`** прикладного сервера.  
4. Проверить лог: сегменты поступают, сообщения уходят на прикладной уровень.

---

## 6. Развёртывание агентного уровня

1. Установить виртуальное окружение Python, зависимости из `requirements.txt` команды.  
2. Задать URL транспортного **POST** приёма сегментов.  
3. Настроить период генерации **3–5 с**, размер сегмента **X**.  
4. Запустить `uvicorn main:app --host 0.0.0.0 --port <порт>` (пример).

## 7. Контроль работоспособности

| Проверка | Действие |
|----------|----------|
| Прикладной REST | `POST /receive` из Postman с телом JSON (поле `username` обязательно) |
| WebSocket | подключение к `ws://<host>:8001/?username=test` |
| Транспорт → прикладной | логи транспорта на успешный HTTP 200 от прикладного сервера |
| Сквозной тест | запущены все три уровня; в UI идёт телеметрия |

---

## 8. Резервное копирование и журналирование

Для учебного стенда достаточно хранить **версии конфигураций** (docker-compose, `.env`, изменённые исходники) в системе контроля версий. Логи сервисов перенаправляйте в файлы для разбора инцидентов на защите.

---

## 9. Устранение неполадок

| Проблема | Действия |
|----------|----------|
| `Transport level unavailable` в ответах | проверить IP/порт транспорта; сетевую связность; firewall |
| Нет сообщений в Kafka | статус брокера; переменные подключения продюсера |
| CORS / смешанный контент | для прод-сборки настроить HTTPS/wss согласованно |

---

## 10. Безопасность

Учебная система не предназначена для эксплуатации в открытом Интернете. Не публикуйте порты Kafka и административные интерфейсы наружу без ACL и TLS.
