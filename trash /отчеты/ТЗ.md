# ТЕХНИЧЕСКОЕ ЗАДАНИЕ  
на разработку распределённой информационной системы обмена данными в реальном времени

**Объект:** система обмена текстовыми сообщениями, файлами и телеметрией с разделением на прикладной, транспортный и агентный уровни.  
**Вариант исполнения:** телеметрия — **вариант 6** (квадрокоптер, видеосъёмка местности; задаётся область съёмки координатами; транслируются положение, высота, остаток памяти для съёмки).  
**Номер варианта по журналу (R):** _____ (используется для вероятности потери сегментов на транспортном уровне, %).

## 2. Цель и назначение системы

**Цель:** спроектировать и реализовать трёхуровневую распределённую систему, обеспечивающую доставку структурированных JSON-данных от источника (агентного уровня) до пользовательского интерфейса (прикладного уровня) с промежуточной буферизацией и сборкой сообщений на транспортном уровне.

**Назначение:** операторский контроль телеметрии БПЛА при условной видеосъёмке заданного участка местности; обмен текстовыми сообщениями между операторами; демонстрация устойчивости к частичной потере сегментов и фиксации ошибки доставки на стороне клиента.

---

## 3. Требования к функциям по уровням

### 3.1. Прикладной уровень

1. Реализовать пользовательский интерфейс на **React**, управление состоянием — **Redux Toolkit**, HTTP-клиент — **Axios**, компоненты — **MUI** (в соответствии с методическими указаниями кафедры).
2. Интерфейс должен отображать окно чата с возможностью ввода текстового сообщения и отправки файла (при наличии соответствующей реализации на цепочке сервисов).
3. При подключении пользователь вводит **имя пользователя**; имя передаётся при каждом сообщении; устанавливается соединение **WebSocket** с сервером прикладного уровня.
4. При обновлении страницы **история чата не сохраняется** (без персистентного хранилища истории на клиенте).
5. Кнопка «Выйти»: очистка чата и сессии пользователя, **закрытие WebSocket**.
6. Если сообщение содержит **признак ошибки** (`error`), текст полезной нагрузки и файлы **не отображаются**; в интерфейсе выводится индикация ошибки.
7. Для варианта 6: ввод **области съёмки** двумя угловыми точками (`lat1`, `lon1`, `lat2`, `lon2`); отображение поступающей телеметрии: **широта, долгота, высота (м), остаток памяти для съёмки (%)**.
8. Метод **Send** (от клиента к серверу прикладного уровня по WebSocket): JSON с отправителем, временем отправки, полезной нагрузкой (текст/файл/поля варианта).
9. Метод **Receive** (от сервера к клиенту): JSON с отправителем, временем, признаком ошибки, полезной нагрузкой.
10. REST **POST /receive** на прикладном сервере принимает собранное транспортным уровнем сообщение и рассылает подключённым WebSocket-клиентам (контракт — OpenAPI `WebSocketServer/openapi.yaml`).

### 3.2. Транспортный уровень

1. Принимать **сегменты** от агентного уровня; каждый сегмент содержит: полезную нагрузку (фрагмент данных), время отправки (идентификатор логического сообщения), общее число сегментов, номер текущего сегмента.
2. Сегменты помещаются в **очередь** (рекомендуемая реализация — **Apache Kafka** + сервис на **Go** или иной согласованной стек-команде).
3. Каждые **N** секунд выполняется попытка **сборки** итогового сообщения и передачи на прикладной уровень (HTTP **POST** на эндпоинт приёма прикладного сервиса).
4. Если за **1–3 цикла** не поступали новые сегменты для незавершённого сообщения, на прикладной уровень передаётся сообщение с **признаком ошибки** (неполная сборка).
5. Перед помещением сегмента в очередь транспортный сервис **намеренно отбрасывает** сегмент с вероятностью **R %** (R — номер варианта по журналу).
6. Обеспечить описание API (**Swagger/OpenAPI**) транспортного сервиса и диаграмму взаимодействия с агентным и прикладным уровнями.

**Исполнитель (транспортный уровень):** ФИО ________________, задачи: очереди, таймер сборки, вероятностная потеря, вызов `/receive`, Docker/конфигурация брокера.

---

### 3.3. Агентный уровень

1. Разбивать полезные данные (телеметрия, текст, метаданные варианта) на сегменты размером **X байт** (значение X согласуется с командой и фиксируется в документации).
2. Передавать каждый сегмент **отдельным HTTP-запросом** на транспортный уровень.
3. Для варианта 6: периодически (каждые **3–5 с**) формировать пакеты телеметрии: текущие координаты, высота, остаток памяти; при необходимости — эмуляция полёта в заданной области съёмки. Рекомендуемый стек: **FastAPI**, развёртывание агента возможно в **ВМ** (по усмотрению команды — связка с ROS 2 не обязательна для минимального варианта).
4. Хранение крупных медиа — по необходимости **объектное хранилище** (MinIO и аналоги); для телеметрии варианта 6 основной акцент — потоковые JSON-данные.

**Исполнитель (агентный уровень):** ФИО ________________, задачи: сегментация, клиент HTTP к транспорту, генерация телеметрии варианта 6, Swagger агентского сервиса.

## 4. Состав данных JSON (согласование через все уровни)

Единая схема полей согласовывается командой; базовый контракт прикладного уровня описан в `WebSocketServer/openapi.yaml` (схема `AppliedMessage`). Для варианта 6 обязательно согласование полей:

- идентификатор пользователя: `username`;
- время: `send_time` (ISO 8601);
- область съёмки: `lat1`, `lon1`, `lat2`, `lon2`;
- телеметрия: `latitude`, `longitude`, `height_m`, `memory_left_percent`;
- ошибка доставки: непустое поле `error`.

## 7. Критерии приёмки

1. Все три уровня обмениваются согласованным JSON по описанным интерфейсам.
