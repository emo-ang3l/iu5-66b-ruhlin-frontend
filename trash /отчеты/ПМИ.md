# ПРОГРАММА И МЕТОДИКА ИСПЫТАНИЙ  
распределённой информационной системы обмена телеметрией и сообщениями

**Объект испытаний:** программное обеспечение прикладного, транспортного и агентного уровней (вариант телеметрии 6).  
**Основание:** техническое задание (приложение А), требования методических указаний кафедры ИУ5.

---

## 1. Цель испытаний

Подтвердить соответствие реализации требованиям ТЗ: сквозная передача JSON, сегментация и сборка, моделирование потерь, корректная индикация ошибок в интерфейсе, работоспособность интерфейсов REST и WebSocket.

---

## 2. Условия проведения

- Стенд из **трёх** узлов локальной сети или эмуляция узлов на одном хосте с разнесением портов (фиксируется в протоколе).  
- Версии ПО: указать фактические (**Node.js**, **Go**, **Python**, Docker, Kafka).  
- Инструменты: браузер (Chrome/Firefox), **Postman** или **Insomnia**, вкладка **Network** DevTools, консоль сервисов.

---

## 3. Порядок испытаний и регистрация результатов

Для каждого требования ТЗ выполняется сценарий; результат — «пройдено» / «не пройдено» с описанием отклонения. **Скриншот** (уменьшенный) вставляется в графу таблицы в итоговом документе Word.

### Таблица 1 — Прикладной уровень (клиент и WebSocket-сервер)

| № | Требование ТЗ (суть) | Порядок действий | Ожидаемый результат | Скриншот (№) |
|---|----------------------|------------------|---------------------|--------------|
| 1 | Ввод имени и установка WebSocket | Открыть UI; ввести имя; нажать «Подключить» | Соединение установлено; в консоли сервера — лог подключения | |
| 2 | Отправка текстового сообщения | Ввести текст; отправить в чат | Сообщение уходит в WS; другой клиент (или второй браузер) получает JSON с полями отправителя и времени | |
| 3 | Приём телеметрии варианта 6 | Запустить агент и транспорт; дождаться потока | В UI отображаются latitude/longitude, height_m, memory_left_percent согласно контракту | |
| 4 | Область съёмки lat1..lon2 | Задать координаты; инициировать сценарий (кнопки/команда согласно UI) | В сообщениях присутствуют согласованные поля области | |
| 5 | Ошибка: непустое `error` | Смоделировать неполную сборку на транспорте | В чате **нет** полезной нагрузки; отображается **индикация ошибки** | |
| 6 | «Выйти» | Нажать «Выйти» | WebSocket закрыт; чат и логин сброшены (по ТЗ) | |
| 7 | Обновление страницы | F5 / перезагрузка | История **не** восстановлена | |
| 8 | REST POST /receive | Postman: `POST http://<host>:8001/receive` с телом `AppliedMessage` | 200 OK; клиенты получают рассылку | |
| 9 | Swagger | Открыть `/api-docs` | Документация отображается, схема `AppliedMessage` доступна | |

## 4. Методика обработки результатов

Отклонения классифицируются: **критический дефект** (невозможность сквозного обмена), **значительный** (нарушение ошибки/контракта), **незначительный** (косметика UI). Испытания считаются успешными при отсутствии критических и значительных дефектов либо при их устранении и повторном прогоне с теми же шагами.
