# Минимальные действия “с нуля”: логин → WebSocket → отправка сообщения (сверено с текущим кодом)

Цель документа: описать **самый короткий путь**, чтобы у вас получилось:

1) ввести имя (логин)  
2) открыть WebSocket соединение на прикладной сервер  
3) отправить сообщение  
4) увидеть, что другое подключение получило это сообщение  

Код в репозитории **не меняем**, но примеры и шаги **сверены** с:

- `WebSocketServer/index.ts` (порт `8001`, WS query `username`, рассылка “всем кроме отправителя”, `/receive`)
- `WebSocketServer/openapi.yaml` (схема `AppliedMessage`)

---

## 0) Что нужно заранее (самое важное)

- **Node.js + npm** установлены.
- Вы понимаете, что есть 2 независимых процесса:
  - **Frontend** (UI в браузере)
  - **WebSocketServer** (Express + WebSocket на `localhost:8001`)

Если **WebSocketServer не запущен**, логин/отправка по WS работать не будут.

---

## 1) Минимальная архитектура (на пальцах)

### 1.1. Кто с кем общается

- Браузер (frontend) **подключается по WebSocket** к `WebSocketServer`.
- `WebSocketServer`:
  - принимает сообщения от клиентов по WS
  - **рассылает их другим подключенным** клиентам (кроме отправителя)
  - параллельно пытается переслать сообщение в транспортный уровень (это не обязательно для “минимальной демонстрации”)

### 1.2. Почему “кроме отправителя”

В текущем коде сервер делает рассылку всем пользователям **кроме** `username` отправителя:

- поэтому, чтобы “увидеть доставку”, нужно **2 подключения**:
  - вкладка A: отправитель `operator1`
  - вкладка B: получатель `operator2`

## 2) Запуск (минимально)

### 2.1. Запуск WebSocketServer

1) Откройте терминал в папке `WebSocketServer/`  
2) Поставьте зависимости:

```bash
npm install
```

3) Запустите сервер так, чтобы он слушал `http://localhost:8001`.

Важно: в `WebSocketServer/package.json` пока нет готового `start`-скрипта, поэтому способ запуска зависит от того, как вы запускаете TypeScript/JS у себя. Главное условие успеха — в логах:

- `Server started at http://localhost:8001`

4) Проверьте Swagger:

- `http://localhost:8001/api-docs`

Если Swagger открылся — значит HTTP-сервер точно поднялся.

### 2.2. Запуск Frontend

В корне репозитория:

```bash
npm install
npm run dev
```

Откройте адрес, который покажет Vite (обычно `http://localhost:5173`).

## 3) Минимальная логика “логин → открыть WS”

### 3.1. Что считается “логином” в минимальном варианте

Логин — это просто строка `username`, которую вы:

- храните в состоянии UI (хоть `useState`, хоть Redux)
- подставляете в URL подключения WebSocket

### 3.2. Какой URL использовать (строго по вашему серверу)

Подключение описано и в коде, и в OpenAPI:

```text
ws://localhost:8001/?username=operator1
```

Смысл параметра `username`:

- сервер по нему группирует подключения в коллекции `users`
- сервер использует его как “имя отправителя”, если вы не передали `username` в JSON

### 3.3. Минимальные обработчики на стороне клиента (что обязательно)

Чтобы понимать, что происходит, на клиенте нужны хотя бы:

- `ws.onopen` — понять, что подключение установлено
- `ws.onmessage` — получить сообщение и показать в UI/консоли
- `ws.onclose` / `ws.onerror` — видеть, что что-то пошло не так

Без `onmessage` вы “не увидите доставку”, даже если сервер всё рассылает.

---

## 4) Минимальная логика “ввод текста → отправка сообщения”

### 4.1. Какой формат JSON можно отправлять (минимум)

По факту `WebSocketServer/index.ts` сервер ожидает строку JSON и делает `JSON.parse`.

Минимальный корректный объект:

```json
{
  "username": "operator1",
  "data": "Привет!",
  "send_time": "2026-03-30T12:00:00.000Z"
}
```

Что реально обязательно:

- для WS: **`data`** (иначе смысла нет) и желательно **`username`**
- `send_time` можно не отправлять: сервер добавит сам, если поля нет

### 4.2. Как отправлять

Логика простая:

1) собрать объект сообщения  
2) `JSON.stringify(message)`  
3) `ws.send(jsonString)`

## 5) Минимальная проверка, что “доставка работает”

Потому что сервер шлёт “всем кроме отправителя”, делаем так:

1) Откройте **две вкладки** фронтенда (или два разных браузера).
2) Вкладка A:
   - логин: `operator1`
   - подключение: `ws://localhost:8001/?username=operator1`
3) Вкладка B:
   - логин: `operator2`
   - подключение: `ws://localhost:8001/?username=operator2`
4) Вкладка A отправляет:

```json
{
  "username": "operator1",
  "data": "Сообщение от operator1"
}
```

Ожидаемое поведение:

- вкладка B получает `onmessage` и отображает “Сообщение от operator1”
- вкладка A **может не получить** это же сообщение обратно (по текущей логике рассылки)

Если вы хотите, чтобы отправитель тоже видел своё сообщение — это делают на фронте (добавляют в локальный список сразу после `ws.send`), либо меняют серверную логику рассылки (но по текущей задаче мы код не правим).

## 6) Альтернативная проверка через Postman: `POST /receive`

Это второй “вход” в прикладной сервер — **со стороны транспортного уровня**.

1) Убедитесь, что WebSocket клиенты подключены (как в пункте 5).
2) В Postman сделайте:

- Method: `POST`
- URL: `http://localhost:8001/receive`
- Body: raw JSON

Пример:

```json
{
  "username": "operator1",
  "data": "Сообщение пришло через /receive",
  "send_time": "2026-03-30T12:00:00.000Z",
  "error": ""
}
```

Ожидаемое:

- Postman получает `200`
- WS-клиенты (кроме `operator1`) получают сообщение

Важно: если в JSON **нет `username`**, сервер вернет `400` (это прямо в `index.ts`).

---

## 7) Минимальный чеклист “если не работает”

### 7.1. Не подключается WebSocket

- Сервер точно запущен на `localhost:8001`?
- Подключаетесь ли вы именно к `ws://localhost:8001/?username=...`?
- Нет ли CORS/прокси? (для WS обычно это не CORS, но порт/адрес должен быть доступен)

### 7.2. Подключается, но сообщения не приходят

- Проверяйте, что **есть 2 разных пользователя** (иначе сервер не отправит обратно отправителю).
- Проверьте, что отправляете **валидный JSON** (иначе `JSON.parse` упадет).
- Проверьте, что на получателе есть `onmessage`.

### 7.3. `/receive` возвращает 400

- В body нет `username` или он пустой.

## 8) Что дальше (после минимального “завелось”)

Когда минимальная цепочка “логин → WS → send → receive” работает, следующий шаг — “привести под вариант 6”:

- в UI сделать карточки телеметрии
- обрабатывать `error` (показывать ошибку вместо данных)
- отправлять команды области (`lat1/lon1/lat2/lon2`) в формате `AppliedMessage`

Источник истины для полей — `WebSocketServer/openapi.yaml`.
