## 1. Введение: что такое “прикладной уровень” тут

В этой работе “прикладной уровень” — это слой между:

- **Frontend (операторская панель / UI)**: показывает телеметрию, статусы, логи, отправляет команды.
- **Транспортный уровень** (отдельный сервис, не в этом репозитории): собирает/доставляет сообщения и дергает REST ручки.

Прикладной уровень делает две ключевые вещи:

1. Держит **WebSocket-подключения** от UI (реальное время).
2. Принимает сообщения по **HTTP** от транспорта и **рассылает** их подключенным WS-клиентам.

В этом репозитории прикладной уровень представлен папкой `WebSocketServer/`.

## 2. Структура репозитория (что где лежит)

- `src/` — фронтенд (React + Vite).
  - Здесь уже есть UI и Redux-персист для пользователя (`src/store/store.ts`), но **WebSocket-клиента в `src/` сейчас нет** (по факту проекта).
- `WebSocketServer/` — сервер прикладного уровня (Express + ws).
  - `index.ts` — **исходник** сервера (главный файл, который нужно читать/править).
  - `index.js` — **сборочный артефакт** (скомпилированный JS). Его не правят вручную.
  - `openapi.yaml` — контракт и документация API.

---

## 3. Что является “истиной” (контракт): `WebSocketServer/openapi.yaml`

Файл `WebSocketServer/openapi.yaml` описывает:

- **HTTP API**: `POST /receive`
- **WebSocket подключение**: `ws://{host}:{port}/?username={имя}`
- **схему данных** сообщений: `AppliedMessage`

Это важно, потому что:

- фронтенд и транспортный уровень должны **сойтись на одинаковых именах полей**;
- любые несоответствия в JSON (например, другое имя `username`/`send_time`) приводят к “тихим” багам (ничего не отображается, или рассылка не работает).

## 4. WebSocketServer: как устроен и что делает

Сервер поднимает **один HTTP-сервер** и на него “вешает”:

- Express-роуты (REST)
- WebSocketServer (`ws`)

Ключевые части `WebSocketServer/index.ts`:

### 4.1. Конфигурация адресов/портов

В начале файла зашиты константы:

- `port = 8001`, `hostname = 'localhost'` — где слушает прикладной сервер.
- `transportLevelHostname` и `transportLevelPort` — куда он пробует отправлять сообщение на транспортный уровень (HTTP `POST /send`).

Если вы запускаете всё локально или в другой сети — эти значения надо менять.

### 4.2. Модель сообщения

В коде используется интерфейс `Message` (аналог `AppliedMessage` из OpenAPI):

- `username` — **обязателен** как идентификатор “от чьего имени” идет сообщение
- `data` — текст
- `send_time` — время (если клиент не прислал — сервер подставит текущий ISO)
- `error` — строка ошибки (если не пустая — UI должен показывать ошибку)
- + поля телеметрии/области (в OpenAPI они перечислены: `lat1/lon1/lat2/lon2`, `latitude/longitude/height_m/memory_left_percent`)

### 4.3. Коллекция подключенных пользователей

Сервер хранит подключенных клиентов в структуре вида:

- ключ: `username`
- значение: массив соединений `{ id, ws }`

Почему массив, а не одно соединение?

- один и тот же пользователь может открыть чат в двух вкладках/браузерах;
- тогда у него будет **несколько** WS-соединений под одним `username`.

### 4.4. WebSocket: подключение (`connection`)

При подключении сервер:

- читает `username` из query-параметра `/?username=...`
- кладет соединение в `users[username]`
- подписывается на:
  - `message` — входящие сообщения от клиента
  - `close` — корректно удаляет только “это” соединение из массива

### 4.5. WebSocket: входящее сообщение (`message`)

На входе сервер получает текстовый кадр, ожидает там JSON и делает:

1. `JSON.parse`
2. гарантирует `message.username` (берет из тела или из query `username`)
3. гарантирует `message.send_time` (если нет — ставит `new Date().toISOString()`)
4. делает **две параллельные операции**:
   - **рассылка** всем остальным WS-пользователям: `sendMessageToOtherUsers(...)`
   - **отправка на транспортный уровень**: `sendMsgToTransportLevel(...)` (через `axios`)

Важно: в текущей реализации отправка на транспортный уровень — “best effort”:

- если транспорт недоступен/вернул не 200 — сервер ставит `message.error`, но **обратно в WS он это сообщение не возвращает** (только логирует).
  - значит, “доставка ошибки отправителю” на UI сейчас не гарантирована: это нужно реализовывать отдельно, если требуется.

### 4.6. HTTP: `POST /receive`

Это входной шлюз **со стороны транспортного уровня**.

Сценарий:

- транспорт собрал итоговое сообщение и дергает `POST http://localhost:8001/receive`
- прикладной сервер:
  - проверяет, что `username` есть (иначе 400)
  - рассылает сообщение всем WS-клиентам, кроме `username` (как и в WS-рассылке)
  - отвечает 200

### 4.7. Swagger / OpenAPI

Сервер публикует документацию:

- `GET /openapi.yaml` — отдает YAML
- `GET /api-docs` — Swagger UI

Это полезно, чтобы быстро проверить, какие поля ожидаются и какие примеры сообщений валидны.

## 5. Потоки данных (кто кому и что шлёт)

### 5.1. UI → WebSocketServer → другие UI (realtime)

1. UI открывает WS: `ws://localhost:8001/?username=operator1`
2. UI отправляет JSON (команда/чат/телеметрия)
3. Сервер получает `message`, парсит JSON
4. Сервер рассылает это всем *остальным* подключенным пользователям (всем `users[key]`, где `key !== username`)

### 5.2. UI → WebSocketServer → Transport level (HTTP)

Одновременно с рассылкой по WS сервер делает:

- `POST http://{transportLevelHostname}:{transportLevelPort}/send` с тем же телом сообщения

Это нужно, чтобы транспортный уровень мог:

- сохранять/агрегировать/передавать дальше в агентный уровень
- реализовывать гарантии доставки (если они есть по ТЗ)

### 5.3. Transport level → WebSocketServer (`POST /receive`) → UI

1. Транспорт вызывает `POST /receive` (на прикладной сервер)
2. Сервер рассылает сообщение WS-клиентам (кроме отправителя по `username`)
3. UI получает `onmessage` и отображает карточку/лог/телеметрию

## 6. Формат сообщений (AppliedMessage) и обязательные поля

С точки зрения сервера **критично** только одно:

- **`username` обязателен** для `/receive`

Для WS-сообщений:

- если `username` не пришел в JSON — сервер подставит его из `/?username=...`, но это работает только если query задан.

Рекомендуемый минимальный формат сообщения (текст/чат):

```json
{
  "username": "operator1",
  "send_time": "2026-03-30T12:00:00.000Z",
  "data": "Сообщение в чат"
}
```

Формат телеметрии (вариант 6) — примеры есть в `openapi.yaml`:

```json
{
  "username": "operator1",
  "send_time": "2026-03-30T12:00:00.000Z",
  "latitude": 55.751244,
  "longitude": 37.618423,
  "height_m": 120.5,
  "memory_left_percent": 78
}
```

Формат ошибки:

```json
{
  "username": "operator1",
  "send_time": "2026-03-30T12:00:00.000Z",
  "error": "Сегменты сообщения не получены в срок"
}
```

## 7. Что без чего не работает (зависимости и деградация)

### 7.1. Что обязательно, чтобы “вообще работало”

- **Node.js + npm** — чтобы ставить зависимости и запускать процессы.
- **`WebSocketServer`** — чтобы UI мог получать realtime по WS и чтобы транспорт мог дергать `/receive`.

Если `WebSocketServer` не запущен:

- UI не сможет подключиться по WS
- транспорт не сможет доставить сообщения через `/receive`

### 7.2. Что “желательно”, но система частично работает без этого

- **Транспортный уровень (`transportLevelHostname:transportLevelPort`)**

Если транспортный уровень выключен/недоступен:

- WS-чат/телеметрия между WS-клиентами продолжат работать (рассылка внутри `WebSocketServer` есть)
- но “проброс” сообщений дальше через `sendMsgToTransportLevel` будет падать в catch и логироваться

Важно: текущая реализация не гарантирует, что UI узнает об ошибке отправки на транспортный уровень.

### 7.3. Swagger/OpenAPI

Swagger **не нужен** для работы WS/REST, но критически полезен для:

- проверки контракта полей
- отладки через Postman/Swagger UI

Если `openapi.yaml` отсутствует или путь неправильный — `/api-docs` не поднимется, но WS/`/receive` могут работать (в зависимости от того, падает ли сервер на чтении файла при старте).

## 8. Как запускать и проверять локально

### 8.1. Запуск фронтенда (Vite)

В корне репозитория:

- установить зависимости: `npm install`
- запустить dev-сервер: `npm run dev`

### 8.2. Запуск WebSocketServer

В папке `WebSocketServer/`:

- установить зависимости: `npm install`
- дальше есть два варианта:
  - если у вас настроен запуск TS (например, через `ts-node`), запускайте `index.ts`
  - если вы запускаете JS — запускайте `index.js`

В этом репозитории `package.json` у сервера пока содержит только `typecheck`, поэтому “скрипт запуска” может быть не прописан — это нормально, но запуск нужно настроить под вашу среду (главное: поднять процесс, который слушает `localhost:8001`).

Признак успеха: в логах будет

- `Server started at http://localhost:8001`

### 8.3. Проверка Swagger

Откройте:

- Swagger UI: `http://localhost:8001/api-docs`
- YAML: `http://localhost:8001/openapi.yaml`

### 8.4. Проверка `/receive` через Postman

POST `http://localhost:8001/receive`, body raw JSON:

```json
{
  "username": "operator1",
  "data": "Тестовое сообщение",
  "send_time": "2026-03-30T12:00:00.000Z",
  "error": ""
}
```

Ожидаемое:

- ответ `200`
- если есть подключенные WS-клиенты (кроме `operator1`) — они получат это сообщение

## 9. Частые ошибки и где искать причину

### 9.1. “WS подключился, но сообщений нет”

Проверьте:

- что клиент подключается с `/?username=...`
- что клиент действительно делает `ws.send(JSON.stringify(obj))`
- что отправляемый JSON валидный (иначе `JSON.parse` на сервере упадет)

### 9.2. “POST /receive возвращает 400”

Причина по текущему коду:

- в body нет поля `username` или оно пустое

### 9.3. “Swagger не открывается”

Проверьте:

- существует ли `WebSocketServer/openapi.yaml`
- что сервер стартует из директории/окружения, где `__dirname` указывает на папку с `openapi.yaml`

### 9.4. “Транспортный уровень недоступен”

Признаки:

- в логах сервера будет `Transport level unavailable` или `Transport level returned non-200`

Это не ломает WS-рассылку, но ломает “проброс” в транспорт.
