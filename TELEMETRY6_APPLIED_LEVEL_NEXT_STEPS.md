## Цель работ (что должно получиться к концу)

Прикладной уровень (`React + Redux Toolkit + Axios + MUI`) для варианта “Телеметрия вариант 6 (квадрокоптер: видеосъёмка местности)” должен:

1. Подключаться к транспортному уровню через **WebSocket** по кнопке “Подключить”.
2. Отправлять команды “задать область съёмки” и “стоп миссии” через WS.
3. Принимать от транспортного уровня телеметрию (раз в 3–5 секунд) и отображать это в виде **чата**:
   - позиция (lat, lon)
   - высота
   - остаток памяти

4. Кнопка “Выйти”:
   - закрывает WebSocket
   - очищает чат
   - очищает логин/имя пользователя
5. История чата **не должна сохраняться** при обновлении страницы (без `localStorage`/`sessionStorage`).
6. Визуально чат и карточки должны соответствовать “дроновой панели” (темная тема, аккуратные карточки, прогресс/цвета для памяти).

## Шаг 2. Привести каркас проекта к “Прикладному уровню по ТЗ”

Сейчас у вас один `App.tsx` и два компонента (Header/Menu).

Минимально вам нужно организовать архитектуру под Redux и WS:

1. Создать store Redux:
   - `src/store.ts` или `src/app/store.ts`
   - root reducer
2. Создать slices (минимальный набор):
   - `connectionSlice`: `isConnected`, `wsStatus`, ошибки соединения
   - `chatSlice`: список сообщений (телеметрия) + `username`
   - `missionSlice` (опционально): `currentStatus`, `currentHeight`, `memoryLeft`, `lastTelemetryTime`
3. Сделать хуки:
   - `useAppDispatch`, `useAppSelector` (или аналог)

В Redux state храните:
- `username`
- `messages[]` (каждое сообщение: `sender`, `time`, `error`, `payload`)
- `connection` (connected/disconnected)
- `lastTelemetry` (последние поля для статуса миссии)

## Шаг 3. Реализовать WebSocket-слой (ключевой функциональный блок)

Создайте сервис/модуль, например:
- `src/services/websocket.ts`

Что он должен делать:

1. `connect(username)`:
   - создает `WebSocket(url)`
   - привязывает обработчики:
     - `onopen`: обновляет Redux `isConnected=true`
     - `onmessage`: парсит JSON и пушит сообщение в `chatSlice`
     - `onerror/onclose`: обновляет `isConnected=false`, при необходимости показывает сообщение об ошибке
2. `disconnect()`:
   - закрывает ws
   - сбрасывает состояние (чат + логин) через action’ы Redux (либо из UI-слоя)
3. `send(message)`:
   - отправляет JSON через WS

Важно по ТЗ:
- username должен передаваться с каждым сообщением.
- при `Exit`:
  - закрыть ws
  - очистить чат
  - очистить логин

## Шаг 4. Прописать типы сообщений (чтобы потом не “сломаться” на JSON)

Согласно ТЗ, входящие сообщения из транспорта/агента имеют общую форму:

```json
{
  "sender": "Drone_01",
  "time": "2026-03-20T14:35:47Z",
  "error": false,
  "payload": {
    "position": {"lat": 55.7558, "lon": 37.6173},
    "height": 120,
    "memory_left": 2.7
  }
}
```

Вам нужно описать TypeScript интерфейсы (минимум):
- `IncomingMessage` (sender, time, error, payload)
- `TelemetryPayload6` (position, height, memory_left) — реальные имена полей возьмите из вашего Swagger/контракта

Также отдельно описать “команды оператора” (исходящие):

1. `set_area` (после ввода lat/lon):
   - payload содержит координаты прямоугольника/области
2. `stop`:
   - payload содержит `{ command: "stop" }` или аналогичное поле из контракта

Поля команд должны соответствовать вашему Swagger (иначе агент/транспорт не поймут).

## Бонус: как подготовить демонстрацию на защите (коротко)

   - открытие WebSocket-соединения
   - исходящие сообщения `set_area` / `stop`
   - входящие сообщения телеметрии (частота 3–5 секунд)
2. Показать интерфейс:
   - чат с обновлением карточек
   - изменение индикации памяти (цвет/прогресс)
   - пример ошибки `error=true` (если в транспортном уровне искусственно теряются сегменты — хотя бы один “красный” кейс)

## Уточняющие поля, которые нужно взять из вашего Swagger/контрактов

Так как в ТЗ прямо сказано “поля называются по вашей теме в Swagger”, вам нужно сверить и после этого точно подставить:

1. Имена полей для `set_area` (какое точное имя у координат/массива/команды)
2. Имена полей в payload телеметрии (например: `memory_left` vs `free_storage` и т.д.)

Чтобы не зависнуть на несоответствии контрактов: открывайте Swagger вашего сервисного контракта и фиксируйте структуру JSON.
